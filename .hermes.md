# Digital Experience Recovery Agent — Project Instructions

## Project Boundary

This repository is the project workspace for the Digital Experience Recovery Agent: a customer-experience and digital-channels recovery product for diagnosing, explaining, and coordinating service recovery during digital banking incidents.

This is not a generic chatbot project. The final product is a self-demonstrating recovery cockpit that shows its own value by running a realistic recovery scenario through Hermes-backed diagnosis, evidence correlation, recovery planning, communication drafting, and human approval gates.

Do not begin implementation until corpus freeze is marked complete.

Current rule:

- If `docs/project-corpus-lock.md` says the build gate is closed, do not write application code.
- While the build gate is closed, only create or update documentation, requirements, schemas, synthetic data specifications, architecture notes, risk registers, and planning artifacts.

## Required Runtime Architecture

Hermes is the single LLM runtime for this project.

- All LLM reasoning, prompt execution, agent-role behavior, tool selection, dossier synthesis, and communication drafting must route through Hermes.
- Do not introduce a second LLM runtime or external agent framework as the reasoning authority unless the corpus explicitly changes this rule.
- Deterministic project tools may retrieve, normalize, validate, or mock data.
- If Hermes support for a desired capability is not confirmed, mark it as `Unspecified`; do not invent support.

## Architecture Conventions

These architecture conventions define how this project should be planned, designed, implemented, and verified.

Use a phased, evidence-first architecture:

1. Corpus and requirements phase
   - Inventory source documents.
   - Extract requirements.
   - Identify missing materials and assumptions.
   - Keep implementation blocked until corpus freeze opens the build gate.

2. Design phase
   - Define product boundaries, data contracts, integration contracts, trust gates, and approval workflows.
   - Separate what Hermes decides from what deterministic tools execute.

3. Synthetic demonstration phase
   - Use synthetic incident data before real data.
   - Demonstrate the recovery cockpit with a reproducible scenario.

4. Implementation phase
   - Begin only after corpus freeze/build gate approval.
   - Keep increments small, tested, and reversible.

5. Verification phase
   - Validate behavior with tests, sample runs, and documented evidence.

Required conceptual roles:

- Digital Experience Supervisor
- Customer Signal Agent
- Observability Agent
- Change Correlation Agent
- Recovery & Communication Agent

These roles must be implemented as Hermes-managed reasoning flows, prompts, subagents, or structured workflows, not as separate LLM runtimes.

## Work Splitting and Subagents

Hermes must split work into phases and subagents when useful.

Use phases when a task mixes discovery, design, implementation, verification, and reporting.

Use subagents or delegation when work can be safely parallelized, such as:

- corpus extraction,
- course-to-project alignment,
- requirements analysis,
- risk review,
- schema review,
- documentation QA,
- test planning.

The final answer must synthesize subagent results and verify shared artifacts directly before declaring completion.

## Coding Standards

Do not touch application code until corpus freeze is marked complete.

When implementation is later allowed:

- Prefer clear, small modules over large hidden frameworks.
- Keep LLM prompts, deterministic tools, data schemas, synthetic datasets, and UI components separated.
- Validate all external inputs.
- Treat customer text and logs as untrusted input.
- Mask or tokenize PII before LLM reasoning.
- Keep side effects behind explicit approval gates.
- Do not hardcode secrets, tokens, internal URLs, customer data, or credentials.
- Store secrets only in approved profile `.env` or secret-management mechanisms.
- Use explicit types/schemas for incident records, telemetry findings, change candidates, recovery recommendations, and communication drafts.
- Keep synthetic data obviously synthetic.

## Test Requirements

Do not create application tests until implementation is allowed.

When implementation begins, tests must cover:

- corpus-derived data schemas,
- synthetic incident ingestion,
- customer signal clustering logic,
- telemetry correlation logic,
- change correlation scoring,
- low-confidence/manual-review routing,
- PII masking/tokenization,
- prompt-injection resistance for untrusted customer text,
- human approval gates for rollback and customer communication,
- incident dossier generation,
- UI/cockpit display states if a UI is implemented.

No feature is complete until it has been exercised with real test commands or a reproducible synthetic run.

## Trust, Privacy, and Safety Standards

- Be cautious with customer data.
- Use synthetic or approved sample data unless a data policy authorizes otherwise.
- Do not ingest real customer data without explicit approval and a privacy/retention policy.
- Do not send customer-facing messages without explicit approval for that specific send.
- Do not execute rollback, deployment, infrastructure, notification, or external side effects without explicit approval.
- Keep an audit trail for incident reasoning, tool inputs, tool outputs, generated recommendations, approvals, and final actions.
- State uncertainty explicitly when evidence is missing or conflicting.

## UI/UX Intent

The final product should feel like a recovery cockpit, not a chat page.

The cockpit should make the incident state easy to understand:

- customer impact summary,
- affected journey,
- symptom clusters,
- telemetry evidence,
- recent change candidates,
- confidence level,
- recommended recovery action,
- communication drafts,
- approval status,
- audit trail.

Design intent:

- calm incident-room interface,
- evidence-first layout,
- high signal-to-noise ratio,
- clear uncertainty labels,
- obvious human approval gates,
- readable customer-facing scripts,
- accessibility-friendly structure that does not rely on color alone.

Dashboard and remote UI capabilities remain `Unspecified` until explicitly designed and verified.

## Build Gate

Implementation is blocked until corpus freeze is marked complete.

Before implementation begins, confirm:

- corpus freeze status is open or missing materials are explicitly waived,
- Session 11/rubric/final instructions are resolved or waived,
- data policy is defined,
- sensitive/internal domains are blocklisted or waived,
- API/dashboard exposure mode is decided,
- required integrations are marked as real, mocked, pending, or unspecified.

If these conditions are not met, continue with documentation, analysis, and design artifacts only.
